들어가며

오브젝트 | 조영호 4장 읽으면서 학습하고 이해한 내용을 나의 언어로 공유합니다.

객체 데이터 중심은 객체의 행동보다는 상태에 초점을 맞추게되어 캡슐화 위반을 쉽게 하게 만든다.

데이터 중심의 설계는 본질적으로 너무 이른 시기에 데이터에 관해 결정하도록 강요한다.

협력에 의한 문맥안에서 객체들간에 역활과 책임을 통해 변경에 유연한 객체지향 프로그래밍과 달리 데이터 중심으로 객체를 바라 보게 되면 무슨 일이 일어 날까요? 필요한 데이터가 무엇이지? 에서 출발 하다 보니 객체 안에 서로 다른 많은 객체를 품을 수 있습니다. 그렇게 되면 그 객체 안에서 데이터를 사용하기 위한 로직이 들어가게 되고 그것은 변경에 취약한 코드를 만드는 셈이죠. 또 getter, setter도 많이 추가 될 수 있어 캡슐화를 위반하기 쉬운 구조가 되는 것입니다. 다시 말하지만 캡슐화를 잘 해야 변경에 유연한 코드를 만들 수 있습니다.

데이터 중심보단 책임(행동) 중심으로 객체를 설계해야 하는 이유

  • 협력하는 그 문맥에서 객체들끼리 하는 역활과 책임이 무엇인가? 에서 출발해야 객체를 설계 할 때, 역활과 책임을 생각하면서 자기가 할 일은 스스로 처리하고 협력하기 위한 인터페이스를 제공함으로써 자연스럽게 캡슐화를 하는 행동으로 이어질 수 있습니다. 데이터는 나중 얘기입니다. 설계의 중심은 내부가 아닌 외부에서 결정해야 합니다. 데이터로 설계하면 그것은 이미 객체를 구현한 것입니다. 객체간에 협력이나 책임에 대한 고려는 없습니다. 협력에 대한 고려없이 협력하려하면 많은 예외 코드가 발생하고 의존도도 높아지게 됩니다.

  • 데이터 중심으로 설계하다보니 데이터를 처리하기 위한 오퍼레이션이 인터페이스(메소드, 파라미터 등)에 고스란히 드러나게 되고 이는 변경에 굉장히 취약해 집니다. 구체적으로 메소드에 파라미터를 3개 받았는데 1개만 받도록 변경되면 어떡하죠? 그 메소드 호출한 코드도 다 변경해줘야합니다.

데이터 중심 설계로인한 결합도가 높아진 케이스

데이터 중심으로 객체를 설계하다보니 가져다 쓰는 데이터 들이 많고 이는 한 객체에 너무 많은 제어 로직을 가질 수 있게 됩니다.
아래 구조도와 같이 ReservationAgency 객체는 너무 많은 객체들에 참조가 이루어져 있고 뭐 좀 변경하려 하면 ReservationAgency 객체도 수정해줘야 합니다. 즉, 겹할도가 엄청 높아인것이죠.
